# ID、Snowflake和精度丢失

想到一个很经典的面试题：
```
UID应该使用int还是string？
```

以前对这个问题无感，因为没遇到过相关问题，现在遇到了，决定记录一下。

## 主键ID和UID
一些系统中，主键ID就是UID，并且UID递增，这个时候UID在数据库就只能使用BIGINT（int64）来做递增了，DAO层获取的也是int64的UID。

一些设计中，会把主键ID和UID分开，主键ID是int64，UID可以string也可以int64，为了方便索引，以及减少存储空间，一般也是存为int64。

## Snowflake
Snowflake算法，用来生成全局唯一ID（int64）。其组成如下：
0. 符号位（1位）固定为0
2. 时间戳（41位）
3. 工作节点ID（10位）
4. 序列号（12位）
这保证了Snowflake算法生成的ID是递增的，理论上不会重复。

## 精度丢失
这次用了Snowflake生成UID，因此最开始理所当然地用了int64做UID，不仅是存储，传输也是全部int64。

但是在前端，诡异的事情发生了，前端从一个端点拿到UID后，直接发给另一个端点，UID居然不同。

通过观察可发现，Snowflake生成的UID几乎占据了int64的全部位，但是JS并没有int64，只有number类型（底层为float64），因此，JS在处理UID的时候，将其存储为了float64，导致了严重的精度丢失。

如果只是计算，精度丢失在误差范围内就无伤大雅。但是全局标识符是不允许丢精度的。所以让AI给我全部换成了string传输。
